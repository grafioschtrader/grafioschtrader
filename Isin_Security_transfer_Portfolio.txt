## Problem
We have the following two problems or use cases:
### New ISIN
- If the ISIN of the security is changed, I cannot simply change it on the existing security. Instead, the following procedure must be followed:
1. Create the same security with this new ISIN if it does not already exist. The settings for the two securities will probably be similar. The two dates “Trade from” and “Active until” for the two securities must form a continuous period.
2. The client sells their open positions on this security with the old ISIN. If possible, the closing price on the day the ISIN was changed should be used. 
3. Purchase the security with the new ISIN. The bank account where these transactions were carried out will have a balance of 0 after these two transactions. This means that the amounts of these two transactions cancel each other out.

### Securities account transfer
- A client has several accounts and wants to transfer a security from one account to another. They must proceed as follows:
1. They must sell the security in the source account. If possible, at the closing price on the day on which the account transfer takes place.
2. They must transfer the cash amount generated by the sale to a bank account of the target portfolio.
3. They must repurchase this security in the target securities account. This use case also does not change the balance of the cash account.

## Solution
### New ISIN
We introduce something like an event table. 
1. The admin role records this renaming of the security. The old and new ISINs and the date of the change must be recorded. When the event is saved, the security is also created under the new ISIN. The “Active until” date for the security with the old security remains unchanged. In the same action, the clients affected by these changes are notified via the internal mail system. In this internal mail, they are asked to apply the event to their affected securities. The event table should also have a running counter that indicates how many clients are affected. The counter adjusts according to the actions performed.
2. Any user can apply this event to their securities if they have a security that falls under this event. The steps listed above will be carried out by the system. Several transactions may be affected by this ISIN change and must update their securities reference. The affected transactions should receive an entry on the reference for the event. Once referenced, an event cannot be changed or deleted. Referencing the event should make it possible to undo this process. This means that a restoration should be possible.

### Event table
I have named the table Event. The definitive and correct name has yet to be found. So far, it knows the two events ISIN change and securities account transfer. The first is a system-wide event and the second is a client-related event. A TreeTable is very suitable for display in the front end; this functionality should be listed in the navigation tree under Basic Data. The name for this functionality still needs to be found. The top hierarchy of this TreeTable would be system- and client-related events. The changes made would be displayed on the second level of the client. The second level of the system would display, for example, a specific ISIN renaming of a security. At this level, the user could then use the context menu to make a corresponding change to their transactions. The client level would then display this change as a child node.

The securities account transfer functionality is executed in a source securities account via the context menu. The corresponding dialog displays the security, source securities account, and the editable two selects for the target portfolio and the target securities account. Of course, the source securities account cannot be present in the securities account options. If there are several securities accounts in a portfolio, the source portfolio must also be specified as the target portfolio.
